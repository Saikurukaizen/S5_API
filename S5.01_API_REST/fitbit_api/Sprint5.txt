API service RESTful - 6 guías básicas se deben cumplir:
- Uniform Interface (UI): Multiple arquitectural constrains are required to guide the behaviour of components.
Resources should be unique through a single URL

- Client/Server based: HTTP response between both. The UI separates user concerns from data storage concerns.
- Stateless operations: Request from client to server must contain all info necessary, so that the server can understand
and process it accordingly. The server can't hold any info about the client state.

- RESTful resource caching: Data within a response to a request must be labeled as cacheable or non-cacheable.

- Code on demand: REST APIs download and execute code in the form of applets or scripts. For example, xml or json

Hay dos caminos para elegir construir una arquitectura RESTful API en Laravel: Usando Sanctum o Passport

Sanctum en un paquete para la autentificación basado en tokens. Se centra en escenarios de autentificación persistente,
mientras que los tokens de corta vida son ideales para autentificaciones temporales (Mobile, SPA Auths)

También ofrece protección CSRF para los usos de autentificación y autorización en las APIs, centrándote en el enfoque de
API endpoints.

Passport ofrece un set completo de endpoints OAuth2 (CONTEXTO)
OAuth2 es un protocolo / framework de autorización que delega la autentificación para aplicaciones web, mobile apps y programas
de escritorio, que dan acceso limitado a las cuentas de usuario en determinados servicios de terceros, que gestiona las cuentas. De
modo que sea éste quien otorgue el acceso.

Roles presentes en OAuth:
	- Client: Aplicación cliente que quiere acceder a la cuenta
	de un usuario en un servicio determinado. Debe contar con una autorización
	del usuario, y ésta se debe validar (a través de la API del servicio)
	- Resource Owner: Quien autoriza a una app para que pueda acceder a su cuenta.
	El acceso está limitado en función del "scope" que concede el usuario durante
	la autorización.
	- Resource Server: Servidor que almacena las cuentas de los usuarios
	- Authorization Server: Verifica la identidad de los usuarios y emite access
	tokens a la aplicación cliente

Además, la arquitectura REST cuenta con 4 RESTricciones

- Respetar la conexión clients/servidor, envio por GET del cliente y POST del servidor
- Los recursos DEBEN y pueden que no, pero deberían ser CACHEABLES
- UI
	- Los recursos deberían ser identificados como URIs en la UI
	- Los recursos deberían ser manipulables mediantes sus representaciones, que es la forma que
	toma cuando alguien lo solicita, lo manipula o lo ve. Un cliente debería poder modificar el recurso mismo (sombras)
	- Los recursos deberían ser autodescriptivos
	- HATEOAS: El motor o estado de la interfaz debería ser hipermedia (el cliente no debería necesitar información adicional
	para navegar por el sitio web) Se puede agregar al servidor un endpoint que devuelva un JSON que informe al cliente
	de cuales son las acciones disponible sobre los recursos que existe en el servidor.
	Autocontenidos y autodescriptivos. Aunque un JSON no es un RESTful, es más sutil, y que una respuesta sea hipermedia, es
	que contener datos, puede contener punteros que direcciona a otro recurso. De esta manera, ese JSON respeta el
	protocolo/standard HAL (Es una forma de hacer un JSON RESTful) Todos los recursos están descritos, y si el cliente está
	utilizando ese mapa JSON, para utilizar la API, si llegase a cambiar la estructura del mapa, va a ser totalmente recibiente
	a estos cambios.
- Stateless: El servidor no debería guardar el estado de la app.
Si una API no es stateless, según y de acuerdo a la Disertación de Roy T Fielding, no sería RESTful
- Los sistemas deberian tener capas y un cliente no debería saber por donde pasa su solicitud
- Code On Demand: Cada vez que vas a un sitio, descargas archivos JS, y los cargas

	Logica:
	Los datos se almacenan en MySQL usando Eloquent, no se necesita almacenar en JSON.
	Cuando recupero datos con eloquest, ej: $discipline = Discipline::find($id), se
	obtiene un objeto o colección de modelos. DisciplineResource transforma ese modelo en
	un array asociativo, y Laravel convierte automáticamente ese array en JSON cuando
	devuelves el Resource desde el controlador.

	Ej: return new DisciplineResource($discipline);

	Donde ocurre la conversión? 
		El Controlador recupera el modelo y lo pasa al Resource
		EL Resource define el array asociativo
		Laravel convierte el Array del Resource a JSON al enviar la respuesta HTTP

---RESPUESTA IA SOBRE EL ARRAY JSON EN KEY/VALUE?


1. ¿Dónde crear el array asociativo “key/value”?
En la respuesta de los controladores:
El array asociativo se crea normalmente en los métodos de los controladores, donde defines la estructura de la respuesta JSON para el frontend.

<?php
return response()->json([
    'message' => 'Disciplina creada',
    'data' => $discipline
], 201);

Esto garantiza que la API siempre devuelva datos en formato JSON, estándar y robusto.

En los recursos (Resource classes):
Para APIs más complejas y mantenibles, puedes usar Resource classes (app/Http/Resources).
Ejemplo:

<?php
return new DisciplineResource($discipline);

Dentro de la clase DisciplineResource, defines el array asociativo que será transformado a JSON.

2. ¿En qué tipo de archivo y dónde debe estar?
Controladores:
Si la lógica es simple, puedes definir el array directamente en el controlador (Api).
Resource classes:
Para lógica más compleja, crea un archivo en app/Http/Resources/DisciplineResource.php.
Esto te permite centralizar la transformación de datos y mantener el controlador limpio.
3. ¿Por qué no en modelos o migraciones?
Modelos:
Los modelos (Models) representan la estructura y relaciones de la base de datos, no la forma en que se devuelven los datos.
Migraciones:
Las migraciones (migrations) solo definen la estructura de las tablas, no la lógica de respuesta.
4. Conclusión y buenas prácticas
Controladores:
Úsalos para lógica sencilla y respuestas rápidas.
Resource classes:
Úsalas para APIs que requieren flexibilidad, personalización y consistencia en las respuestas.
Nunca en migraciones o modelos:
Mantén la lógica de presentación separada de la lógica de datos y estructura.
Recomendación:

Para mantener tu API escalable y robusta, crea Resource classes en app/Http/Resources y usa arrays asociativos en ellas para definir la estructura JSON de tus respuestas.
Los controladores pueden usar estas resources para devolver los datos al frontend.
¿Quieres un ejemplo de cómo crear y usar un Resource en Laravel para tus endpoints?

2. Estructura del archivo app/Http/Resources/DisciplineResource.php:

<?php

namespace App\Http\Resources;

use Illuminate\Http\Resources\Json\JsonResource;

class DisciplineResource extends JsonResource
{
    public function toArray($request)
    {
        return [
            'id'          => $this->id,
            'name'        => $this->name,
            'description' => $this->description,
            'community'   => new CommunityResource($this->community), // si tienes relación
            'users'       => UserResource::collection($this->users),  // si tienes relación
            // Puedes añadir más campos o lógica aquí
        ];
    }
}

3. Usar el Resource en el DisciplineController:

<?php
use App\Http\Resources\DisciplineResource;

public function show($id)
{
    $discipline = Discipline::findOrFail($id);
    return new DisciplineResource($discipline);
}

public function index()
{
    return DisciplineResource::collection(Discipline::all());
}

 #[Test]
    public function it_returns_percentage_of_users_per_discipline(): void
    {
        $this->actingAsAdmin();
        $disciplines = Discipline::factory()->count(3)->create();
        $userCounts = [5, 3, 2];
        foreach ($disciplines as $i => $discipline) {
            User::factory()->count($userCounts[$i])->create(['discipline_id' => $discipline->id]);
        }
        $totalUsers = array_sum($userCounts);
        $expectedPercentages = [];
        foreach ($disciplines as $i => $discipline) {
            $expectedPercentages[$discipline->name] = round($userCounts[$i] / $totalUsers * 100);
        }
        $response = $this->getJson('/api/stats/disciplines/stats');
        $response->assertStatus(200);
        foreach ($expectedPercentages as $name => $percentage) {
            $response->assertJsonFragment([
                'name' => $name,
                'percentage' => $percentage
            ]);
        }
    }

    #[Test]
    public function it_returns_ranking_of_disciplines_by_users(): void
    {
        $this->actingAsAdmin();
        $disciplines = Discipline::factory()->count(3)->create();
        $userCounts = [2, 6, 4];
        foreach ($disciplines as $i => $discipline) {
            User::factory()->count($userCounts[$i])->create(['discipline_id' => $discipline->id]);
        }
        $ranking = collect($disciplines)
            ->map(function ($discipline, $i) use ($userCounts) {
                return [
                    'name' => $discipline->name,
                    'users' => $userCounts[$i]
                ];
            })
            ->sortByDesc('users')
            ->values()
            ->all();
        $response = $this->getJson('/api/stats/disciplines/stats');
        $response->assertStatus(200);
        foreach ($ranking as $rank) {
            $response->assertJsonFragment($rank);
        }
    }
